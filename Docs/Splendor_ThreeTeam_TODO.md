# Splendor 三组分工 ToDo（网络+交易 / 玩家逻辑 / UI+卡牌+领主）

## 使用说明
- 本文按三条主线分工，每条主线由 1 名主负责人牵头。
- 每项任务都要求基于服务端权威与 NetworkVariable/NetworkList 同步。
- 每组需在每日站会同步风险和阻塞，跨组依赖写在任务末尾。

---

## 第一部分：网络 + 交易逻辑（负责人 A）

### 目标
完成完整联机交易闭环：请求、服务器校验、状态同步、失败反馈。

### ToDo
1. 固化交易动作入口：拿代币、买卡、预约、还币都走“客户端发请求 -> 服务器校验 -> NetworkVariable/NetworkList 同步”。
2. 完成拿代币规则的全量校验：同色拿2、异色拿3、库存不足、非当前回合、还币锁状态。
3. 完成银行库存管理：按 2/3/4 人动态初始化库存，服务器唯一写入库存状态。
4. 完成交易失败反馈链路：服务器失败原因可回传到对应客户端 UI 提示。
5. 完成市场交易广播：卡牌被买走或预约后，统一触发市场更新事件。
6. 完成联机一致性检查：同一操作在 Host/Client 端显示一致，无本地分叉。
7. 建立回归测试用例：至少覆盖拿代币、买卡、预约、超10还币四条主流程。

### 验收标准
1. 任意客户端无法直接改资源数据，只有服务器改动有效。
2. 两个以上客户端联机时，库存、卡牌状态、玩家资产全量一致。
3. 非法请求均被拦截并返回可读错误信息。

### 主要依赖
1. 依赖第三部分提供可触发交易的 UI 入口。
2. 向第二部分提供稳定的回合状态与交易结果事件。

---

## 第二部分：玩家逻辑（负责人 B）

### 目标
完成玩家回合状态机、资产变化和胜负判定逻辑。

### ToDo
1. 完成玩家资产逻辑：基础宝石、黄金、永久折扣、分数全在服务器侧更新并同步。
2. 完成回合推进状态机：动作完成后统一推进到下一玩家，断线时安全跳过。
3. 完成超10代币还币逻辑：强制锁操作、提交还币、还币后解锁并推进回合。
4. 完成预约位规则：最多三张预约卡，买下预约卡后正确清槽。
5. 完成终局规则：达到 15 分后触发“本轮结束”流程并统一结算。
6. 完成排名规则：分数优先，若平分按开发卡数量等规则决胜。
7. 与领主系统对齐：买卡后触发领主判定，领主加分计入终局排名。

### 验收标准
1. 回合顺序在多人联机下稳定，不出现卡回合。
2. 还币锁期间所有非法操作都被拦截。
3. 终局时所有客户端同时进入结果状态且名次一致。

### 主要依赖
1. 依赖第一部分提供可靠交易结果与回合权限控制。
2. 依赖第三部分提供终局展示与玩家面板刷新。

---

## 第三部分：UI 和卡牌，领主（负责人 C）

### 目标
完成可操作、可观测、可反馈的完整前端表现层（含领主）。

### ToDo
1. 完成主界面交互入口：拿代币、买卡、预约、还币按钮和输入面板。
2. 完成市场卡牌 UI：三层明牌展示、可买状态高亮、不可买遮罩。
3. 完成牌堆状态 UI：显示各层剩余张数，并在补牌后自动刷新。
4. 完成玩家面板 UI：代币、折扣、黄金、分数实时刷新。
5. 完成领主面板 UI：显示当前明置领主、需求条件、被拿走后及时移除。
6. 完成领主反馈：玩家满足领主时显示明确提示，支持后续动画扩展。
7. 完成错误提示通道：交易失败、非法操作、资源不足都要有可读提示。
8. 完成联机可视一致性验证：两个客户端界面展示一致，刷新无明显延迟。

### 验收标准
1. UI 不直接修改权威数据，只发请求并显示同步结果。
2. 卡牌、牌堆、领主三类面板均可稳定刷新。
3. 玩家能在 UI 层清晰理解“为什么能做/不能做某个动作”。

### 主要依赖
1. 依赖第一部分提供网络事件与服务器返回结果。
2. 依赖第二部分提供玩家状态、回合状态、终局状态。

---

## 联调节奏建议
1. 第 1 周：第一部分和第二部分先打通拿代币/买卡/回合切换。
2. 第 2 周：第三部分接入牌堆与领主面板，完成基础联机可玩。
3. 第 3 周：三组联合做终局结算、异常流程和回归测试。

## 风险清单
1. 客户端本地逻辑过重导致与服务器状态不一致。
2. 事件订阅与反订阅不完整导致重复触发或内存泄漏。
3. 断线重连后 NetworkVariable 初始状态未正确重建。
